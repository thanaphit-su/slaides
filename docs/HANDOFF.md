# SLAIDES — Handoff to Implementation

You — the next agent or human engineering team — own getting SLAIDES from prototype to running v0.1. This file tells you what to build, in what order, and where each design decision was already made.

The historical **prototype** (`.draft/prototype/slaides.html` + `.draft/prototype/src/*.jsx`) is a visual reference, not the current implementation source of truth. Current behavior lives in `apps/web` and `apps/api`; the **specs** in `docs/SPEC.md`, **requirements** in `docs/REQUIREMENTS.md`, and **architecture** in `docs/ARCHITECTURE.md` define behavior, contracts, and infra. **Design tokens** are in `docs/DESIGN.md`.

**Widget AI phase usability test checklist (manual):**
1. **Behavior clarification:** Open editor -> widget sidebar create mode -> ask `Create an audience mood widget` without saying Quiet/Loud/shared/private. Expected: no behavior picker is visible; if AI is uncertain it asks a clarification question; option chips appear above the chat input and choosing one continues the workflow.
2. **Loud creation without downgrade:** Ask `Create a live audience poll with shared results`, insert it, then preview/live-test the widget. Expected: saved widget behavior remains Loud and audience contribution works; it must not save as Quiet.
3. **Quiet -> Loud adjust:** Start from a Quiet widget, open AI Adjust, ask `Change this to a loud widget where audience responses are tallied live`, and apply. Expected: behavior becomes Loud, a new revision appears, and the live contribution path works.
4. **Loud -> Quiet adjust:** Start from a Loud widget, ask AI Adjust `Make this private per viewer and remove shared audience state`, and apply. Expected: behavior becomes Quiet, shared contribution/state code is removed or unused, and a new revision appears.
5. **Thread history reload:** Open AI Adjust, send a change request, apply or leave the draft visible, close/navigate away, then reopen AI Adjust for the same widget. Expected: previous user messages, AI plan/reflection/draft/apply history are still visible.
6. **Agent progress rendering:** Ask for a complex adjustment such as `Refactor this widget so it supports configurable labels and explain your plan first`. Expected: `plan`, `step`, or `reflection` responses render as workflow progress and do not show "AI response was not a valid widget workflow."
7. **Example props preview:** Generate a props-driven widget such as `Create a quiz widget with question text and three choices as props`. Expected: preview/thumbnail/new placement uses realistic sample values instead of an empty-looking widget, and the Props tab starts from useful defaults/examples.
8. **Placement revision stability:** Attach a widget showing visible content "Version A", adjust the widget to "Version B", then inspect the existing placement/session history path. Expected: the placement can still render the captured original revision where revision snapshots apply; later edits do not silently rewrite history.
9. **Copy safety:** Copy a widget from another deck or duplicate it in the same deck, insert the copy, and adjust it. Expected: copied widget has its own current revision and adjusting the copy does not mutate the original.
10. **Create, leave, return, adjust:** Generate and insert a widget, refresh or leave/reopen the editor, open AI Adjust, then ask `Based on the current widget, make it more compact`. Expected: AI has enough stored spec/thread context to revise the widget, and applying creates another revision.

`make supabase-status` prints Docker Compose service status plus the local API, Studio, and Postgres endpoints. Supabase local keys + the HS256 `JWT_SECRET` live in [docker/supabase/.env.slaides](../docker/supabase/.env.slaides); the backend auto-loads them (including `SUPABASE_JWT_SECRET`, used for local JWT verification — see the 2026-05-25 502 fix above) when the corresponding SLAIDES env vars are empty and `DATABASE_URL` points at `localhost:54322`.

**Op note — root `.env` `JWT_SECRET` collision.** Docker Compose auto-interpolates `${JWT_SECRET}` in the included supabase docker-compose from the project `.env` next to it (i.e. the **root** `.env`), *before* `env_file: docker/supabase/.env.slaides` is applied to container env. If the root `.env` defines a different `JWT_SECRET` (the SLAIDES API's own jwt-secret slot also reads that name), the running Supabase auth container ends up with the wrong signing secret — `SERVICE_ROLE_KEY` fails verification, `make seed`'s admin-create silently 403s, and the seed user's password never gets set, so sign-in returns 401. **Keep the root `.env`'s `JWT_SECRET` equal to `.env.slaides`'s value** (the demo string `super-secret-jwt-token-with-at-least-32-characters-long` for the vanilla local stack) — or remove it from root `.env` entirely (it only fell back into `llm_encryption_secret` historically, which can be set explicitly via `LLM_ENCRYPTION_SECRET`).

**DATABASE_URL talks to the pooler, not raw Postgres.** Port `54322` on the local Supabase stack is Supavisor (session mode). It rejects bare `postgres` usernames with `Tenant or user not found`. Use `postgresql+asyncpg://postgres.<POOLER_TENANT_ID>:postgres@localhost:54322/postgres` — the tenant id is set to `slaides` in `.env.slaides`, so the working DSN is `postgresql+asyncpg://postgres.slaides:postgres@localhost:54322/postgres`.

`make test` is intended to run backend + frontend tests. In this workspace the verified commands are `uv run pytest` from `apps/api` (57 tests) and `npm test` from `apps/web` (51 tests). If `make test` uses a system Python without `fakeredis`, prefer the `uv run pytest` command.

Realtime over `wss://…/ws/sessions/{id}?token=…` with Redis pub/sub fanout (channel `sess:{id}`) — slide advances broadcast in <500 ms, questions land in the presenter's right rail within 1 s, audience votes round-trip into deck widgets via `vote.broadcast`, and native live poll/question/random events update `session_slide.spec/results`. Presence keys `sess:{id}:presence:{ref} EX 30` are heartbeated by the audience socket every 10 s, and a `beforeunload`/`pagehide` hook closes the WS cleanly on tab close so the server drops presence + publishes `participant.left` immediately. Presenter and audience deck-slide views both support read-only selected-text interpretation through an AI-only toolbar and use matching non-slim slide typography. Presenter navigation uses the editor-style bottom stepper with clickable dots/count and supports ArrowLeft/ArrowRight for previous/next. Audience navigation uses the same sticky stepper style for presenter-passed slides only, supports ArrowLeft/ArrowRight within that passed history, and keeps Raise Question as a lower-right floating `?` button. The Open Interaction FAB opens `LiveInteractionSheet`; presenters can launch poll, open-question, or random-audience `session_slide`s. New live interactions insert immediately after the current presentation point and auto-advance audiences via `slide.changed`. Presenters can edit interaction specs, pause/reopen voting, reset polls, promote/unpromote/hide open-question answers, save live polls/questions as deck widgets, and view random-audience picks sampled from active participants. Audience poll votes update `session_slide.results` and replay for late joiners; open-question answers stream host-only as `question_answer.new` until promoted, then render as featured cards for the audience; normal audience questions use a toast confirmation and Enter-to-send instead of a blocking success dialog. Audience cap of 500 (configurable via `SESSION_AUDIENCE_CAP`) enforced in `POST /auth/guest` with a 409. Ending a session broadcasts a `session.ended` event that redirects both presenter and audience out of the live surfaces.
- **LLM.** Workspace LLM settings are stored server-side: base URL, encrypted API key, a model library (`llm_models` stored inside workspace JSON), and capability-to-model routing (`inline_write`, `interpret`, `widget_generate`; `None` disables that capability). Each model can carry advanced parameters (`max_context_window`, `max_output_tokens`, `temperature`, `top_p`, `frequency_penalty`, `presence_penalty`) and a `supports_image_input` flag; there is intentionally no provider-name or internet-search setting. `POST /api/v1/llm/complete` streams OpenAI-compatible chat completions as SSE (`token`, `done`, `error`), resolves the model/parameters from the purpose's assigned capability, accepts image inputs only when that model is flagged image-capable, and logs calls to `llm_call` with a prompt hash, not raw prompt text. Editor selected text can be sent to an Interpret popover, the context menu exposes **Interpret with AI**, live presenter/audience selected text can be interpreted from `SlideStage`, and instructor/widget host contexts can call `window.slaides.api.llm()` through the iframe bridge. Audience widget LLM calls are intentionally rejected; audience live interpret calls are allowed with guest JWTs and only for the active session workspace.

### Resolved QA fixes (2026-05-20)

These issues were reported during the QA pass and are now fixed in source:

1. **Slide insertion / sections.** Add Slide Above and Add Slide Below now pass the active slide's `section_id`. Empty sections have an explicit Add slide affordance, and section header `+` appends into that section via `editor.insertSlideInSection`.
2. **Markdown editing UX.** The right-click "Edit as Markdown" modal has been removed. The editor now uses a visible Rendered / Markdown segmented control, matching GitHub/IDE-style mode switching. Markdown mode edits the canonical slide markdown inline.
3. **Markdown renderer / contenteditable.** Empty slides render an editable paragraph, unordered and ordered lists parse/render/serialize as lists instead of collapsed paragraphs, GitHub-style markdown tables parse/render as real tables, direct `# `/`## `/`### ` shortcuts promote paragraphs to headings, native Enter is normalized after the browser mutates contenteditable DOM, and Cmd/Ctrl+Z resyncs markdown state after browser undo.
4. **Deck delete persistence.** Deck deletion now runs through `delete_deck()` and deletes dependent slides before sections, while nulling historical runtime/session/widget references as needed to avoid FK violations.
5. **Widget UI.** The right widgets drawer is persistent and collapsible; Generate with AI is a chat thread; Adjust uses the same sidebar in adjust mode; widget source is editable through the sidebar Code tab with explicit Save; widget chrome has hover/focus-only bottom-right icon-only Adjust and Remove controls.
6. **Live audience access and layout.** Ended sessions reject guest auth, audience snapshots, and WS auth; direct `/audience/:sessionId` requires the correct stored guest token and redirects on failure. The audience presenting view is now full-width responsive on desktop while staying compact on mobile.
7. **Live AI interpretation + LLM settings.** Presenter and audience live deck slides now support selected-text interpretation through a read-only AI-only toolbar. Guest interpret calls are scoped by the guest JWT and cannot invoke widget generation. LLM settings now support multiple model IDs, advanced per-model parameters, capability-to-model routing, and image-capable widget-generation models; the old visible transcript-summary capability and provider-name field are gone.

### Important decisions taken during M0–M4

M0+M1:
- **Auth history.** M0 used local argon2 + JWT for instructor auth to stay self-contained. As of 2026-05-22, instructor auth is Supabase Auth; only audience guest tokens remain custom SLAIDES JWTs.
- **Local database history.** M0 used a standalone `postgres:16-alpine` service. As of 2026-05-22, normal local development uses the vendored Supabase self-hosted compose stack with Postgres 17 exposed on `54322` so Studio can inspect/edit SLAIDES tables. The legacy Postgres compose service and `legacy-postgres-up` target were removed.
- **Portable `GUID` SQLAlchemy type** (Postgres native UUID + SQLite `CHAR(36)`) so the test suite can use SQLite tempfiles without docker. Pattern stays for M3+ tables.
- **Server returns truth, client reconciles.** The frontend optimistically updates local state on slide edits; the PUT response returns the canonical `slides[]` and the editor store reconciles via `mergeSlides`. M3's WebSocket pipeline follows the same rule. (Originally the PUT endpoint also auto-split slides on additional H1s; that behavior was removed — user typing never spawns new slides, and `split_on_h1` is retained as a utility for future paste/import flows. New slides are created via the Add Slide ribbons / `POST /decks/{deck_id}/slides`.)
- **Newly-inserted/updated rows are refreshed after flush** so server-default timestamps (`created_at`, `updated_at`) populate before Pydantic serialisation. See [decks/service.py](../apps/api/src/slaides/decks/service.py). Skipping this breaks `from_attributes` validation.

M2:
- **Widget placeholders are markdown's responsibility, mounting is the renderer's.** The markdown subset still treats `{{widget:placement_id}}` as a single block. The renderer ([markdown/render.ts](../apps/web/src/markdown/render.ts)) accepts `widgets`, `getWidget`, `WidgetFrameComp`, `onAdjust` and mounts the iframe **inline** at the placeholder position. `serialiseContentEditable` reads `data-widget-id` off the wrapper and re-emits the placeholder verbatim — typing around a widget never destroys it. The wrapper is `contenteditable="false"` so the iframe is interactive while the rest of the slide stays editable.
- **Props are baked into the srcdoc, not negotiated via handshake.** [WidgetFrame.vue](../apps/web/src/widgets/WidgetFrame.vue) injects `window.__slaides_boot = {props, role, participant}` as an inline `<script>` *before* the bridge runs, so widget scripts can read `window.slaides.props` synchronously at script-evaluation time. We tried the bridge-handshake-then-init pattern first and it raced with widget init code — every widget would fall back to default props because `bridge.init` arrived after the script's first read. Future widgets should still subscribe to `props` (via `bridge.on('props', cb)`) for live updates from the adjust sidebar.
- **iframes auto-size to content.** The bridge inside the widget measures `body.scrollHeight` after DOMContentLoaded and via `ResizeObserver` + `MutationObserver` (coalesced through `requestAnimationFrame`), and posts `{type: "resize", payload: {height}}` to the host. The host applies the measured height and uses `scrolling="no"` + `overflow: hidden` so the iframe never shows its own scrollbar — the parent slide scrolls instead. No widget is allowed to set a fixed iframe height.
- **1-widget-per-slide is enforced server-side.** `POST /decks/{deck_id}/slides/{slide_id}/widgets` returns 409 if a `slide_widget` row already exists for that slide. The UI hides the `Add a widget` ribbon and uses the right sidebar's adjust mode for the mounted widget in parallel — but the server is the source of truth. Don't relax this without changing both layers.
- **Slide-attach mutates the slide markdown.** Attach appends `{{widget:placement_id}}` to the slide; detach strips the line. This keeps the markdown a complete description of the slide layout (so `.slaides` export round-trips), and it means the editor reload after attach/detach automatically picks up the new placeholder position.
- **`.swidget` format = HTML + JSON frontmatter.** The widget file is a real HTML document. A leading `<!-- slaides-widget {…} -->` comment carries the metadata (name, kind, description, props_schema, tags, version). Body, `<style>`, and `<script>` blocks parse as standard HTML. This means widgets are inspectable in any text editor and can be hand-edited. See [widgets/package.py](../apps/api/src/slaides/widgets/package.py).
- **The seed reconciles widget placements on every run.** Initially the seed only linked widgets when the deck was being created. After adding/changing widget props that left existing rows stale. Now `seed.py` always walks the deck's slides, finds the placeholders referenced in markdown, and `INSERT … ON DUPLICATE`s the `slide_widget` rows (with `widget_id` + `props` refreshed). Pattern for M3+ seeds: never skip-if-exists for things that might evolve.

M3:
- **WebSocket auth via query-string token.** Browsers can't set request headers on `new WebSocket(…)`, so the connect URL carries `?token=…` and the handler decodes either an instructor access JWT (must match `session.owner_id`) or a guest JWT (`sid` claim must match the path session_id). Anything else closes the socket with `1008 Policy Violation`. The same `auth.service.decode` is reused — no separate WS-only signing key.
- **Lazy pubsub subscriber per session.** The `Hub` only starts a Redis subscriber task the first time someone connects, and shuts it down when the last connection leaves. Two `_SessionState` fields hold the subscriber `Task` and its `pubsub` so we can `unsubscribe` cleanly. The race we hit during tests — publishing before the listener subscribed — is fixed by `await pubsub.subscribe(...)` *synchronously inside `register`* before scheduling the listener task.
- **Server-authoritative interactions.** Audience widgets are no longer allowed to optimistically increment local counts; they emit `vote`, the backend persists to `interaction_log`, broadcasts on `sess:{id}`, and the session store re-injects the broadcast into every WidgetFrame via a `slaides:widget-broadcast` `CustomEvent` (translated to `vote.broadcast` in the bridge). This keeps the voter's view consistent with everyone else's at the cost of one round-trip of latency on click. LLM streaming follows the same ownership principle: provider calls run through the backend proxy, not directly from the browser.
- **Native live interaction state lives on `session_slide`.** Poll/question/random specs are stored in `session_slide.spec`, canonical poll tallies/promoted answers/random-audience picks live in `session_slide.results`, and raw votes/answers are tied back through `interaction_log.session_slide_id`. That gives late joiners the current state from the normal audience snapshot without replaying the whole log, while preserving the raw log for moderation and transcripts.
- **Audience snapshot is a separate endpoint.** `GET /sessions/:id` requires the instructor; the audience uses `GET /sessions/:id/audience` with a guest token. We added a `current_principal` dep that accepts either kind, but kept the two surfaces distinct so future audience-only fields (e.g. redacted answers) can diverge without authorization gymnastics.
- **`participant_ref` is the only id in `interaction_log` and `question`.** Cleartext email is stored on `participant` for the session scope only — instructors see "Anonymous" if `anon=True`, otherwise the participant's `display_name`. The hash is `sha256(email.strip().lower() + session.salt)`, so an instructor cannot link refs across sessions even with the same audience email. Per [REQUIREMENTS.md](REQUIREMENTS.md) FR-071 / NFR-033.
- **Live interaction slides inherit the current theme by default.** `session_slide.inverted_theme` remains stored for existing rows and future explicit use, but new FAB-created poll/question/random slides now default to `false` so the presenter and audience stay on the same light/dark presentation surface. The API schema, ORM default, and DB server default all use `false` as of `0007_session_slide_theme_default.py`; existing rows are not rewritten.
- **FAB-created interaction slides are inserted at the current presentation point.** `insert_session_slide_after_current()` anchors new `session_slide` rows to the current deck slide (or the current session slide's parent) and shifts sibling interaction positions, so quick polls/questions/random picks sit immediately after the topic the presenter was on. `POST /sessions/:id/interactions` also broadcasts `session_slide.inserted` followed by `slide.changed`, which is why audience tabs advance immediately without F5.
- **`POST /sessions/:id/advance` is also the broadcast channel.** No separate "advance via WS" path on the server. The host calls REST, the REST handler updates the row, and the same handler calls `broadcast_slide_changed` via the Hub. The frontend mirrors this: `session.advanceTo()` calls REST + optimistically sets `current_slide_id`; the inbound WS event is idempotent. If you add new host-initiated live changes, follow the same shape — REST endpoint + Redis publish from inside the handler.
- **`current_slide_id` is intentionally FK-less.** Originally it was `ForeignKey("slide.id")`, which threw `relation not present` when the FAB inserted a `session_slide` and we advanced to it. The 0004 migration drops the constraint; the column now holds *either* a deck slide id or a session_slide id. The frontend already disambiguates by looking the id up in both lists. If you ever need to join here, do an outer-join against both tables, not a single FK lookup.
- **Audience tabs fetch widget bodies with their guest token.** `GET /widgets/{id}` originally required `current_user`, which 401'd for audience members. It now uses `current_principal` (either an instructor or a guest); for guests it verifies the widget is actually referenced by their session — either as a deck-slide placement or by a `session_slide`. Audiences in different workspaces still can't read each other's widgets. The frontend uses `widgetsApi.getAs(id, guestToken)` in [Audience.vue](../apps/web/src/pages/Audience.vue) which passes the guest JWT explicitly (the api client now respects an explicit `headers.Authorization` over the auth-store token).
- **Vite proxies `/ws` with `ws: true`.** Without this, audience tabs running through the dev server (no `VITE_API_URL` set → WS URL falls back to `ws://localhost:5173/...`) silently failed and looked like "audience never syncs / questions don't send / widgets never load." All three symptoms were one missing proxy entry.

M4:
- **Workspace-level LLM config remains the only config scope.** `PATCH /workspace` updates base URL, encrypted API key, the model library, capability-to-model routing, and advanced model parameters; deck/session/widget-level LLM settings are intentionally absent. `GET /workspace` returns `llm_key_configured`, never the key itself. Model config intentionally has no provider-name field and no internet-search toggle.
- **Capabilities route to models, not booleans in the UI.** The visible capabilities are `inline_write`, `interpret`, and `widget_generate`; the user selects a model id or `None` per capability. The backend keeps compatibility by returning `llm_caps` booleans derived from assignments, but the actual routing source is `llm_capability_models` plus `llm_models`. The older `summarise` purpose is still in the type layer for future transcript work, but it is not a visible Settings capability.
- **Image input is opt-in per model.** `LlmCompleteRequest.images[]` carries data URLs. `_resolve_model()` rejects image inputs unless the routed model has `supports_image_input`. `WidgetCollection` only shows the `+` attachment button when the model assigned to Generate widgets is image-capable.
- **API keys are encrypted at rest.** `llm.crypto` derives a Fernet key from `LLM_ENCRYPTION_SECRET` plus `workspace_id`; if that env var is missing it falls back to `JWT_SECRET` for local development. Production should set `LLM_ENCRYPTION_SECRET` explicitly before users store real provider keys.
- **LLM calls stream only through the backend.** The browser calls `/api/v1/llm/complete`; the backend streams OpenAI-compatible `/chat/completions` chunks back as SSE events. Provider keys never reach the browser or widget iframe. Guest tokens are accepted only for `purpose="interpret"` and resolve the workspace from the live session, not from client-provided context.
- **Prompt text is not persisted.** `llm_call.prompt_hash` records a SHA-256 of the prompt for dedupe/debugging, while `prompt_text` is intentionally `NULL`. If transcript summaries need raw transcript text later, store that in transcript-specific tables, not `llm_call`.
- **Rate limiting runs against Redis.** Defaults are 60 workspace calls/min and 6 widget-generation calls/min/user. `INCR` + `EXPIRE` on epoch-minute bucket keys, shared across uvicorn workers via the same Redis connection the WS hub uses; the keys self-expire so there's no janitor.
- **Widget LLM calls are host-gated.** `window.slaides.api.llm()` is available in the bridge, but [WidgetFrame.vue](../apps/web/src/widgets/WidgetFrame.vue) rejects audience-role requests and only proxies non-audience widget calls through the instructor's authenticated browser session.
- **Widget frame postMessage payloads must be clone-safe.** `WidgetFrame.vue` deep-clones outbound host-to-iframe payloads before calling `postMessage`, including live `props.update` messages. Do not send Vue reactive/proxy objects across the iframe boundary; browsers throw `DataCloneError`, which can abort Vue's update flush and leave unrelated UI stuck until refresh.
- **Frontend SSE parsing is intentionally small.** [api/llm.ts](../apps/web/src/api/llm.ts) handles only the events emitted by our backend (`token`, `done`, `error`) and can pass an explicit token/images payload when needed. Keep the provider-specific streaming parser in the backend service. `done.warnings` is an optional array surfaced via an `onWarnings` callback on `completeText`.
- **Widget theme contract is enforced at three layers.** (1) The iframe `:root` carries the Shadcn-style token set baked from host `getComputedStyle` at mount — this is the single source of truth widgets read via `var(--*)`. (2) The `widget_generate` system prompt (`WIDGET_THEME_CONTRACT` in `llm/service.py`) lists every token, names which font family backs each `--font-*`, and forbids hex/rgb/hsl literals, hardcoded font-family strings, `@import`, `<link rel=stylesheet>`, and remote `<script src=https://...>`. (3) The backend `_scan_theme_violations` scans the assembled stream and surfaces categorized warnings in `done.warnings`; the Adjust chat bubble renders them in an amber notice above the Apply button. Warnings are non-blocking — the user always has the final call. If a future widget genuinely needs a one-off literal color (rare), the user can apply through the warning.
- **AI Adjust apply path is per-field PATCH.** The chat prompt tells the model to "include ONLY fields that change; omit unchanged fields", `parseDraft` only keeps fields the AI explicitly returned, and the apply sends a `PATCH /widgets/:id` with just those keys. This means a one-line "rename to X" edit doesn't risk overwriting the widget's HTML or JS. Frontend `validateDraftHtml` additionally refuses to apply when `<` / `>` counts are unbalanced or when the resulting HTML is suspiciously short (defense against mid-stream truncation).
- **AI Adjust history persistence is non-blocking after apply.** Once the widget `PATCH` succeeds, the UI marks the draft applied and emits `applied`; appending the AI thread "apply" message is fire-and-forget. Keep `void appendAiThreadMessage(...)` in this path so a slow or failed conversation-history write cannot keep the Apply button in `applying` after the widget revision was already committed.
- **Slide canvas repaints on cache mutations via a `widgetRev` counter prop.** `SlideCanvas.paint()` only ran from `slideId` / `widgets` watchers, so out-of-band cache mutations (AI Apply, manual Code-tab Save, library delete) didn't refresh the iframe. The editor now bumps `widgetRev` whenever the cached widget body or membership changes, and `SlideCanvas` watches it. Use the same signal for any future flow that mutates a widget body without changing its slide placement.
- **Widget iframe height must measure content, not viewport.** When the host injects `html,body{height:100%}` so widgets using `height:100%` render, the body's resolved height can equal the iframe's external height. More generally, `documentElement.scrollHeight` / `body.offsetHeight` can reflect the host-assigned iframe height. If the WidgetFrame then bumps its measured height by `ceil(scrollHeight + 2)`, the body resizes, the bridge re-emits, and the iframe grows indefinitely. Fix: in fill mode, ignore resize messages entirely and pin the iframe at `minHeight`; in inline mode, the bridge reports body content height only (`body.scrollHeight` / body rect), and `WidgetFrame` updates only on meaningful height changes with no `+2px` buffer.
- **Widget library deletion uses `force=true` cascade rather than soft-delete.** `DELETE /widgets/:id` without `force` returns 409 with `usage_count` so the UI can show a confirmation modal scaled to the impact. With `force=true` it strips `{{widget:placement_id}}` lines from every affected slide markdown, deletes the `slide_widget` rows, and NULLs the `widget_id` columns on historical `session_slide` and `interaction_log` rows (audit trail survives). Frontend reloads the deck after delete so any active slide that lost its widget repaints without the stale iframe.

## Translation notes (React → Vue 3)

- **State:** `useState` → `ref` / `reactive`. Local state stays in the SFC; cross-page state goes to Pinia.
- **Effects:** `useEffect` → `watchEffect` / `onMounted` / `onUnmounted`.
- **Conditional rendering:** keep it. `v-if` for actually mounted/unmounted, `v-show` for cheap toggles.
- **Lists:** `.map` → `v-for` with `:key`. The prototype's keys (e.g. `slide.id`, `question.id`) carry over directly.
- **Inline styles:** Vue accepts `:style="{}"` the same way. Keep inline for one-offs; promote repeated patterns to scoped CSS or design tokens.
- **Animations:** the prototype uses CSS keyframe utility classes (`.fade-in`, `.slide-up`, `.slide-in-right`, `.scale-in`). Reuse the same classes. For Vue's `<Transition>`, only wrap when an element actually mounts/unmounts.

The harness drove three rounds of additive tuning of `WIDGET_THEME_CONTRACT` in [llm/service.py](../apps/api/src/slaides/llm/service.py):

1. **Round 1** — added the **bridge event conventions** table (`vote`/`vote.broadcast`, `text`/`text.broadcast`, `plotter.update`, …), the **props read pattern** (`var p = (window.slaides && window.slaides.props) || {}` plus `bridge.on('props', cb)` for live updates), and **semantic skeletons** per kind so models stop emitting `<div>` soup. Lifted poll 0.40 → 0.93 average and most seeds to 1.00.

2. **Round 2** — tightened the no-literal-color rule with explicit bullets for backgrounds, hover/active states, border colors, box-shadow colors, and translucent overlays, with the rule "if you find yourself wanting `rgba(...)`, the answer is `var(--accent-soft)`". Closed the last `rgba` slip-up.

3. **Round 3** — added a **canvas / imperative paint** section explaining that `ctx.strokeStyle` / `ctx.fillStyle` need resolved color strings, so the model should `getComputedStyle(document.documentElement).getPropertyValue('--accent').trim()` at script start instead of hardcoding hex in canvas paint code. Belt-and-braces: 3-run plotter baselines were already passing via this pattern, but a 1-run probe surfaced occasional hex slips in canvas APIs.

Reproducibility caveat: the user's OpenAI-compatible router (`Qwen/Qwen3.5-397B-A17B`) emits `delta.reasoning` chunks before `delta.content` and occasionally yields very long `data:` lines that trip `httpx.aiter_lines()` mid-line. The eval harness inlines a tolerant byte-buffer SSE parser (split on `\n\n`, join multi-line `data:` segments, skip non-JSON keepalives) so this is captured cleanly. Production `_stream_openai_chunks` still uses `aiter_lines()` — that's fine for the browser SSE consumer where Fetch handles framing, but worth noting if we ever move provider streaming entirely to backend-orchestrated paths.

## Open questions

Resolved through M4:
- **Auth provider** — instructor auth is Supabase Auth with local approval profiles. Audience guest auth remains a custom session-scoped SLAIDES JWT because it is tied to live-session identity, not instructor accounts.
- **Database host** — normal local development uses Supabase local Postgres so Studio can inspect and edit SLAIDES tables. All schema still goes through Alembic.
- **File storage for `.slaides` / `.swidget` blobs** — kept direct byte streams. The current export endpoints return the zip body inline with `Content-Disposition`; v0.1 blobs are small (markdown + a few KB of widget code) and fit comfortably in a response. Move to signed URLs only if blob sizes grow (e.g. once widget assets are allowed).
- **Widget code editing** — implemented without Monaco. The right widget sidebar Code tab exposes plain HTML/JS/CSS textareas, keeps edits local, and persists only when the user clicks Save. AI-driven Adjust still PATCHes only changed fields after draft validation.
- **Audience capacity per session** — soft cap of 500 enforced in `POST /auth/guest`, configurable via `SESSION_AUDIENCE_CAP`. A real load test before production is recommended but the gate is in place.

Still open (resolve before the corresponding milestone):
- **Widget iframe + LLM origin hardening** (before production M4) — Widget iframes use sandboxed `srcdoc` and `WidgetFrame.vue` now only proxies non-audience LLM calls, but CSP and `postMessage` origin/source checks still deserve a security pass before untrusted widgets can call the LLM proxy.
- **Widgets v2 — property-driven + Loud/Quiet behaviour** — design brief: [WIDGETS_V2.md](WIDGETS_V2.md). Resolves "widgets can only be customised by editing HTML" (Topic 1) and "AI-generated widgets can't share state across audience" (Topic 2). Sequenced as four shippable slices; Step 1 (props panel) lands the biggest UX win and supersedes much of the "Legacy iframe widget result replay" item below — that item is now subsumed under Step 4.
- **Legacy iframe widget result replay** (before richer custom live widgets) — Native `LivePollSlide` now persists tallies on `session_slide.results` and late joiners see current counts from the snapshot. Older iframe-based widgets still aggregate broadcasts locally unless their state is materialized elsewhere. Widgets v2 Step 4 replaces this with a `placement_state` table and a unified `widget.contribute` / `widget.state` protocol.
- **Theme bridge dynamic flip** (before dark-mode wiring in M5) — `WidgetFrame` snapshots host tokens at mount; if the user toggles dark mode mid-session, mounted widgets keep their old palette until re-mount. Either re-emit `:root` via a `theme.update` postMessage when the host class flips, or watch the host root for class changes and `srcdoc`-reload affected iframes.
- **Widget body changes don't broadcast in live sessions** — If an instructor applies an AI edit or manual Code-tab Save while a session is running, audience tabs keep showing the pre-edit widget (they fetched it once on connect). Acceptable for v0.1 since widget editing is authoring, but flag for the live-collab milestone: broadcast a `widget.updated` event on the `sess:{id}` channel and have audience-side `WidgetFrame`s refetch + re-srcdoc.
- **Transcript redaction** (M5) — Do we let the instructor permanently redact a question/answer? Spec it before M5.
- **Cancel in-flight LLM streams** — `llmApi.completeText` accepts an `AbortSignal` but no UI surfaces it; once you hit Send in the Adjust chat you wait. Add a cancel button bound to a per-message `AbortController` before any long-running model is supported.

## Things to NOT do (until v0.2)

- Don't add slide transitions or animation primitives. Fade-in is it.
- Don't add live cursors / presence in the editor.
- Don't expose per-deck LLM config — workspace-level only.
- Don't add a public widget marketplace. The per-workspace library is enough.
- Don't add image/video uploads to markdown. Text-only.
- Don't translate the UI. English-only.

If a feature feels like it'd be nice — and it's not in `docs/REQUIREMENTS.md` — push it to the v0.2 backlog and keep moving. Scope discipline is how v0.1 ships.
